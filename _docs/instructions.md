# Правила разработки

Документ описывает правила, которым обязан следовать разработчик и/или LLM-ассистент при работе над проектом. Дополняет общие гайдлайны из `CLAUDE.md` (think before coding, simplicity first, surgical changes, goal-driven execution) проектными правилами.

## 1. Обязательное чтение перед задачей

Перед любой нетривиальной задачей:

1. Прочитать `CLAUDE.md` (общие принципы поведения LLM-агента).
2. Прочитать `_docs/README.md` — сориентироваться в навигации.
3. Прочитать `_docs/current-state.md` — узнать про известные проблемы и легаси.
4. Прочитать релевантный раздел: для RAG → `rag-pipeline.md`, для MCP → `mcp-tools.md`, для CLI → `commands.md`, для конфигурации → `configuration.md`.
5. Прочитать `_board/plan.md` целиком — понять контекст и зависимости.

При расхождении между документом и кодом — **приоритет у кода**, документ обновляется ближайшим коммитом.

## 2. Принципы внесения изменений

Эти правила берут начало в `CLAUDE.md` и адаптированы под проект:

### 2.1 Сначала думаем, потом кодим

- Сформулировать гипотезу: что меняем и зачем.
- Если есть несколько разумных интерпретаций задачи — спросить, не выбирать молча.
- Если есть более простой способ — озвучить.
- Если что-то непонятно — остановиться и уточнить.

### 2.2 Минимализм

- Никаких фич, которых не просили.
- Никаких абстракций для одного использования.
- Никаких параметров «на будущее».
- Никакой обработки невозможных сценариев.
- Если получилось 200 строк там, где хватило бы 50 — переписать.

### 2.3 Хирургические правки

- Менять **только то**, что нужно для задачи.
- Не «улучшать» соседний код, комментарии, форматирование.
- Не рефакторить то, что не сломано.
- Совпадать со стилем существующего кода, даже если ты бы написал иначе.
- Если заметили мёртвый код, не относящийся к задаче — упомянуть в комментарии PR/коммита, не удалять самостоятельно.

### 2.4 Цель → проверка

Каждая задача должна иметь явный критерий «готово»:

- «Добавить валидацию» → «Написать тест на невалидный ввод, потом сделать его зелёным».
- «Починить баг» → «Написать тест, воспроизводящий баг, потом сделать его зелёным».
- «Зарефакторить X» → «Тесты проходят до и после».

## 3. Стиль кода

### 3.1 Python-стиль

- Python 3.10+. Type hints **обязательны** в публичных функциях/классах. Внутренние утилиты — на усмотрение.
- Имена: `snake_case` для функций/переменных, `PascalCase` для классов, `UPPER_CASE` для констант (включая `config.py`).
- Длина строки — **до 100 символов**, целевое — 88–100.
- Docstrings — кратко, на английском (как сейчас в проекте). Не переводить существующие docstring без явной задачи.
- Никаких `print` для отладки, **за исключением** уже принятой в проекте практики `print(...)` как «лог пользователю» в CLI. Для нового кода — использовать `logging`, как только в проект придёт `logging`-конфиг (см. `roadmap.md`).
- Импорты: stdlib → third-party → local. Не оставлять закомментированных импортов.

### 3.2 Структура импорта `config`

В существующих модулях используется хак:

```python
sys.path.insert(0, str(Path(__file__).parent.parent))
from config import ...
```

**Не повторяйте это в новых модулях**. Лучше:

- если меняешь существующий модуль — оставь как есть,
- если создаёшь новый модуль — поставь импорт `from src.config import ...` (после превращения `src/` в пакет, см. `roadmap.md`),
- либо до рефакторинга — продолжи существующий хак, но **только в `src/rag/*` и `src/mcp/*`**.

### 3.3 Логирование

В коде сейчас `print(...)`. При добавлении нового кода:

- если это «диагностика для CLI-пользователя» (статус сборки индекса, прогресс) — `print(...)` ОК (соответствует существующему стилю).
- если это «программный лог» (ошибки, длительности, аргументы вызовов) — заведи `logger = logging.getLogger(__name__)` и используй его. Это шаг к будущей `logging`-инфраструктуре (см. `roadmap.md`).

Никогда не логировать содержимое документов / запросов в полном виде — это потенциально приватные данные. В крайнем случае — длина и хеш.

## 4. Git-дисциплина

### 4.1 Ветки и коммиты

- Основная ветка — `main`.
- Рабочие ветки — `feature/<краткое-имя>`, `fix/<краткое-имя>`, `task-<id>-<имя>`.
- Коммиты — атомарные, в императиве, на английском (как принято в open-source):
  - `feat(rag): add chunk overlap configuration`
  - `fix(mcp): handle empty arguments in read_document`
  - `docs(architecture): update component diagram`
  - `refactor(assistant): extract decision-prompt into method`
  - `chore(plan): start task 3`
- Префиксы из conventional commits: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `style`, `perf`.

### 4.2 Что нельзя коммитить

- Реальные документы пользователя (`src/docs/*.md`, `*.txt`, `*.pdf`, `*.docx`, кроме `.gitkeep`).
- Артефакты сборки (`*.faiss`, `*.pkl`).
- Виртуальные окружения (`.venv/`, `venv/`).
- Кеши (`__pycache__/`, `.pytest_cache/`, `.mypy_cache/`, `.ruff_cache/`).
- Конфиги IDE (`.idea/`, `.vscode/`).
- Локальные конфиги (`config.local.py`, `.env`).
- Логи (`*.log`, `logs/`).

Все перечисленное уже в `.gitignore` (корневом и `src/.gitignore`). Если случайно закоммитили — переписать историю или вынести в отдельный коммит-удаление.

### 4.3 Размер коммита

Один коммит ≈ одна логическая единица. Желательно:

- < 300 изменённых строк,
- одна затронутая подсистема,
- сообщение однозначно отвечает на «что было сделано».

### 4.4 Комментарии и документация

- Не добавлять и **не удалять** комментарии и docstrings без явной задачи на это.
- При изменении поведения **обязательно** обновить документ в `_docs/`, который этому поведению соответствует.

## 5. Безопасность

- Не делать сетевых вызовов из MCP-сервера или RAG-конвейера. Единственный разрешённый сетевой адрес — `localhost:11434` (Ollama).
- При расширении MCP — сохранить containment: любой инструмент, обращающийся к ФС, должен проверять, что путь внутри `DOCUMENTS_DIR` (через `Path.resolve()`-сравнение).
- Не использовать `pickle.load` для данных, пришедших от пользователя или из недоверенных источников. `chunks.pkl` создаётся самим приложением — это допустимо.
- Не хранить токены/пароли в коде. Если в будущем появятся — через переменные окружения и/или `config.local.py`, который в `.gitignore`.

## 6. Обработка ошибок

- Любой сетевой вызов (`requests.post`, `MCPClient._send`) — потенциальная точка отказа. При добавлении новых нужно:
  - либо ловить и преобразовывать в осмысленное сообщение пользователю,
  - либо явно пропускать наверх до общего обработчика в `main.py`.
- Не глотать исключения молча через `except Exception: pass`. Минимум — логировать.
- В существующем `_llm_decide_mcp_usage` есть `except Exception: return None, None` — это сознательный fallback (если LLM решила непонятно — лучше не использовать MCP, чем упасть). Этот паттерн **не повторять** для новых мест без явной аргументации.

## 7. Производительность

- Загрузка `SentenceTransformer` — медленная (1–3 сек). Не создавать модель внутри функции, вызываемой много раз. Текущий код держит `model = SentenceTransformer(...)` на уровне модуля — соответствует правилу.
- FAISS-поиск дёшев. Не оптимизировать его без замеров.
- Ollama-вызов — самая дорогая часть пайплайна. Не делать его дважды без необходимости. В `assistant.query` сейчас два вызова: decision + final answer — это сознательно.
- Pickle-загрузка `chunks.pkl` происходит при первом импорте `rag.query`. Не вызывать пересборку индекса в hot path.

## 8. Тестирование (текущее и целевое)

### 8.1 Текущее состояние

Автоматических тестов в проекте **нет**. Это известный недостаток (см. `current-state.md`, `roadmap.md`). Большинство правок проверяется вручную в интерактивном режиме.

### 8.2 Что делать сейчас (до появления pytest-инфраструктуры)

При значимой правке:

1. Запустить `python main.py build-index` — индекс должен собраться без ошибок (если есть документы в `src/docs/`).
2. Запустить `python main.py` и задать 2–3 вопроса, проверить:
   - ответы не пустые,
   - источники возвращаются,
   - решение про MCP принимается (попробовать вопрос «list all documents»).
3. Завершить через `exit` — не должно быть `Unclosed subprocess` или похожих варнингов.

Опционально: `python -m rag.query` для проверки чистого RAG.

### 8.3 Целевое состояние (после внедрения pytest)

См. отдельный план в `roadmap.md`. Когда pytest появится, минимальный набор:

- `tests/rag/test_chunk.py` — детерминированность чанкования.
- `tests/rag/test_query.py` — `retrieve` на маленьком тестовом индексе.
- `tests/mcp/test_server.py` — каждый из трёх инструментов (read/list/search).
- `tests/test_assistant.py` — `_llm_decide_mcp_usage` с замоканной LLM (несколько JSON-вариантов).

## 9. Что делать с легаси и багами

- Заметили баг — добавить запись в `_docs/current-state.md` (раздел «Известные проблемы») с минимальным воспроизведением. Не править «попутно», если задача — другая.
- Заметили мёртвый код — то же самое: запись в `current-state.md`, без удаления вне задачи.
- Заметили технический долг — кандидат в `_docs/roadmap.md` (Phase соответствующая).

Это правило берёт начало в `CLAUDE.md` § «Surgical Changes»: «If you notice unrelated dead code, mention it — don't delete it».

## 10. Локальный запуск (dev-цикл)

```bash
# 1. Окружение (один раз)
python -m venv .venv
source .venv/bin/activate
pip install -r src/requirements.txt

# 2. Ollama (один раз)
ollama serve &
ollama pull qwen3:0.6b

# 3. Подготовка (один раз / после смены документов)
cp -r ~/your/docs src/docs/
cd src
python main.py build-index

# 4. Прогон
python main.py
# ...вопрос-ответ-вопрос...
# exit

# 5. После правок кода
python main.py build-index   # если задели rag/* или config
python main.py               # smoke-test нескольких вопросов
```

## 11. Что строго запрещено

- Добавлять облачные LLM/embeddings (OpenAI, Anthropic, Cohere) — противоречит local-first.
- Делать сетевые вызовы за пределы `localhost:11434`.
- Коммитить артефакты (`*.faiss`, `*.pkl`), виртуальное окружение, реальные документы пользователя.
- Делать массовые рефакторинги «попутно» при выполнении задачи на 3 строки.
- Удалять секции из `_docs/*` без обновления связанных разделов и `_docs/README.md` § «Навигация».
- Менять формат decision-промпта в `_llm_decide_mcp_usage` без обновления `mcp-tools.md` § «Decision-протокол».
