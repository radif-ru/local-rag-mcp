# План развития

Документ агрегирует кандидатов в спринты: что улучшать, чинить и расширять. Не является обязательным планом — это меню для составителя ТЗ. Фактический план задач — `../_board/plan.md`.

## Принципы приоритизации

1. **Сначала корректность** (баги из `current-state.md` § «Известные проблемы»).
2. **Потом надёжность** (тесты, таймауты, логирование).
3. **Потом удобство** (рефакторинг, унификация клиентов, переименования).
4. **И только потом фичи** (Web-UI, API, метаданные, реранкинг).

## Использование документа

При составлении нового спринта или ТЗ:

1. Открыть `current-state.md` — приоритизировать критичные баги.
2. Выбрать одну тематическую Phase — не «брать по фиче из каждой фазы».
3. Сформулировать задачи в `_board/plan.md` со статусом `ToDo`, описанием, Definition of Done и зависимостями.
4. Удалить кандидатов, попавших в активный спринт, из этого документа (или пометить «in plan: task N»).

## Размеры и риски (легенда)

- **Объём**: XS (< 1ч), S (1–4ч), M (4–16ч), L (1–5 дней), XL (> 1 недели).
- **Риск**: низкий (понятная задача, не ломает ничего), средний (нужны решения по дороге), высокий (требует дизайн-документа отдельно).
