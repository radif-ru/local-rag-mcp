# Текущее состояние проекта

Документ фиксирует **фактическое** состояние кода: что работает, что криво, где известны баги, где легаси-нюансы. Обновляйте при появлении новых проблем или их закрытии. Это ключевой документ при онбординге и перед любыми правками.

## 1. Что работает

- ✅ **Сборка FAISS-индекса** из документов в `DOCUMENTS_DIR`. Поддержка `.md`, `.txt`, `.pdf`, `.docx`. Команда `python main.py build-index`.
- ✅ **Семантический поиск** — `IndexFlatIP` + L2-нормализованные эмбеддинги (косинусная близость), top-K чанков.
- ✅ **Интерактивный Q&A** — `python main.py` с многострочным промптом и атрибуцией источников.
- ✅ **MCP-сервер с тремя инструментами**: `read_document`, `list_documents`, `search_documents`. Stdio-транспорт, JSON-RPC.
- ✅ **MCP-клиент** в виде subprocess, корректный init и shutdown (`assistant.close()`).
- ✅ **LLM-decision промпт** — модель сама решает, нужен ли MCP-инструмент, в формате строгого JSON.
- ✅ **Защита от path traversal** в `read_document` через `Path.resolve()` + `startswith`.
- ✅ **Авто-сборка индекса при отсутствии** — если `index.faiss`/`chunks.pkl` нет, `_ensure_index_exists` пытается собрать их.
- ✅ **Корректная обработка пустого индекса** — при отсутствии документов `retrieve()` возвращает `[]`, пайплайн идёт по «пустому» шаблону промпта.

## 2. Известные проблемы (баги, недочёты, легаси)

### 2.1 Мёртвый и потенциально падающий код в `ingest.py`

**Файл**: `src/rag/ingest.py:39–50`. **Серьёзность**: средняя (бомба замедленного действия).

```python
for path in base_dir.rglob("*"):
    if path.is_file() and path.suffix.lower() in SUPPORTED_EXTENSIONS:
        ...
    else:
        [documents.append(doc) for doc in ingest_documents(path)]
```

Проблемы:

1. `rglob("*")` уже возвращает все вложенные пути рекурсивно — рекурсивный обход через `else` бессмыслен.
2. `ingest_documents()` в текущей сигнатуре **не принимает аргумент** `path` — попадание в `else` приведёт к `TypeError: ingest_documents() takes 0 positional arguments but 1 was given`.
3. Сейчас `else` не достижим в нормальных условиях (для каталогов условие `path.is_file()` ложно, и они туда попадают, но обработка никогда не «провалится», пока в каталоге не окажется директории с поддерживаемым расширением, что маловероятно).
4. List comprehension с side-effect через `.append` — антипаттерн.

**Рекомендация**: удалить `else`-ветвь полностью или превратить в no-op (continue). См. `roadmap.md` Phase 1.

### 2.2 Несоответствие токенайзера эмбеддинг-модели

**Файл**: `src/rag/chunk.py:9`. **Серьёзность**: низкая (работает, но субоптимально).

`tiktoken.get_encoding("cl100k_base")` — токенайзер от OpenAI. Эмбеддинг-модель `all-MiniLM-L6-v2` использует WordPiece, у LLM Qwen3 — собственный BPE. Размер чанка в `cl100k`-токенах ≠ количество токенов на входе эмбеддера и ≠ количество токенов в LLM-контексте.

**Последствие**: чанки могут не соответствовать «700 токенам с точки зрения эмбеддера» (часто меньше или больше). На качестве поиска это сказывается слабо, на размере промпта — заметно.

**Рекомендация**: либо токенайзер от эмбеддинг-модели (`model.tokenize(...)`), либо посимвольный размер. См. `roadmap.md`.

### 2.3 Автозагрузка/автосборка индекса при импорте `rag.query`

**Файл**: `src/rag/query.py:73`. **Серьёзность**: средняя (медленный старт, side-effect при импорте).

```python
# Initialize index on module load
_ensure_index_exists()
```

Эта строка выполняется **при первом импорте модуля**. Если индекса нет — запускается полная сборка (load + chunk + embed). Старт `python main.py` без индекса может занимать десятки секунд без видимой причины (build-index не вызывался явно).

Дополнительно: модуль `mcp/server.py` НЕ импортирует `rag.query`, поэтому MCP-сервер на это не натыкается. Но `assistant.py` импортирует `rag.query` через `from rag.query import retrieve, build_prompt, ask_llm` — при `from assistant import CompanyKBAssistant` в `main.py` это сразу триггерит автозагрузку.

**Рекомендация**: убрать вызов из module level, делать lazy load в `retrieve()` (который уже умеет вызывать `_ensure_index_exists()` если нужно).

### 2.4 Относительный `DOCUMENTS_DIR`

**Файл**: `src/config.py:4`, использование в `mcp/server.py` и `rag/ingest.py`. **Серьёзность**: средняя (security при неаккуратном запуске).

`DOCUMENTS_DIR = "./docs"` — относительный путь. Безопасность `read_document` зависит от cwd процесса:

- При `python main.py` (через `MCPClient`, который делает `cwd=src/`) — `./docs` указывает на `src/docs/`, всё корректно.
- При `python src/mcp/server.py` (вручную из корня репо) — `./docs` указывает на корневой `./docs`, которого нет, и проверка резолвится в неожиданном направлении.

**Рекомендация**: вычислять абсолютный путь в `config.py` через `Path(__file__).parent / "docs"` либо вообще запрещать запуск MCP-сервера без явного `cwd`.

### 2.5 Отсутствие таймаутов

**Файлы**:
- `src/rag/query.py::ask_llm` — `requests.post(...)` без `timeout=`.
- `src/mcp/client.py::_send` — `proc.stdout.readline()` без таймаута.

**Серьёзность**: средняя. Если Ollama зависнет или MCP-сервер не ответит — клиент висит навсегда. Только `Ctrl+C` спасает.

**Рекомендация**: добавить `timeout=` в `requests.post`, добавить `select`/`asyncio` для чтения stdout с таймаутом. См. `roadmap.md`.

### 2.6 Stderr MCP-сервера не читается

**Файл**: `src/mcp/client.py:14`. **Серьёзность**: средняя (отладка тяжелее).

Subprocess запущен с `stderr=subprocess.PIPE`, но никто не читает этот pipe. При больших объёмах stderr (например, traceback из FastMCP) **pipe-буфер заполнится и subprocess зависнет** — это известный паттерн в Python.

**Рекомендация**: либо `stderr=subprocess.DEVNULL`, либо отдельный поток-читатель stderr с логированием.

### 2.7 Дублирование цикла в `assistant.py` и `main.py`

**Файлы**: `src/main.py:9-66` и `src/assistant.py:167-201`. **Серьёзность**: низкая (дубль кода, но не баг).

Оба файла реализуют почти идентичный CLI-цикл (input → query → print). `main.py` — без rich-вывода, `assistant.py::__main__` — с rich-форматированием markdown. **Один из них должен исчезнуть** или их надо явно разделить (например, `main.py --rich`).

### 2.8 Использование двух разных Ollama-клиентов

**Файлы**:
- `src/rag/query.py::ask_llm` — `requests.post('/api/generate', ..., stream=False)`.
- `src/assistant.py::_llm_decide_mcp_usage` — `ollama.Client().chat(...)`.

**Серьёзность**: низкая.

Это два разных способа дёрнуть одну и ту же Ollama. `ollama` Python SDK более структурирован (типизированные ответы), `requests` — без зависимостей. Унификация сделает код проще.

### 2.9 Отсутствие тестов

**Серьёзность**: высокая в долгосрочной перспективе.

Нет ни одного автоматического теста. Проверка изменений — только ручная. При росте кодовой базы это станет блокером. См. `roadmap.md`.

### 2.10 Отсутствие логирования

**Серьёзность**: средняя.

Везде `print(...)`. Ничего не остаётся в файле, нет уровней (debug/info/warning/error), нельзя выключить «болтовню» в production. Кандидат в `roadmap.md`.

### 2.11 Имя пакета `mcp` пересекается с возможным сторонним

**Файл**: `src/mcp/`. **Серьёзность**: низкая (потенциальная).

Локальный пакет назван `mcp`. Если в будущем появится официальный SDK с именем `mcp` (на PyPI, например, `mcp` от Anthropic) и он попадёт в `requirements.txt`, начнутся конфликты импортов.

**Рекомендация**: переименовать в `mcp_tools` или `kb_mcp` при ближайшем рефакторинге. Сейчас работает, но стоит держать в уме.

### 2.12 Глобальное состояние модуля `rag/query.py`

**Файл**: `src/rag/query.py:22-23`. **Серьёзность**: низкая.

`index = None` и `chunks = []` — переменные модульного уровня, которые мутируются `_ensure_index_exists()`. При импорте из нескольких мест это даст один общий индекс (по сути — синглтон), что не плохо для текущего сценария, но затрудняет тестирование (нельзя инициализировать с альтернативным индексом).

### 2.13 Корневой `.gitignore` агрессивно игнорирует `.md` в `docs/`

**Файл**: `.gitignore:219-222` (корневой).

```gitignore
docs/*.md
docs/*.txt
docs/*.pdf
docs/*.docx
```

Это глоб от любого подкаталога `docs/`. То есть **невозможно закоммитить `_docs/*.md`** — этот документ! Сейчас `_docs/` назван с подчёркиванием специально, чтобы не подпасть под `docs/*`. Но это хрупкий контракт.

**Также проверьте**: строка `/**.md` на строке 243 — это глоб **в корне репо** на любые `.md`. `_docs/README.md` им НЕ подпадает (`/**.md` в `gitignore` означает только корневой уровень). Всё корректно, но запутанно. Кандидат на чистку игнора.

### 2.14 Возможные проблемы с правами на `pickle.load`

**Файл**: `src/rag/query.py:55`. **Серьёзность**: низкая (мини-RCE при компрометации артефактов).

`pickle.load(open(chunks_path, "rb"))` — если злоумышленник заменит `chunks.pkl`, это RCE. Поскольку артефакт создаётся самим приложением и в `.gitignore`, при правильной гигиене проблема не возникает. Но для production с многопользовательской системой стоит мигрировать на JSON/parquet.

## 3. Архитектурные нюансы (не баги, но знать обязательно)

- **Ассистент не хранит диалог** — каждый запрос самостоятельный.
- **MCP-вызов опциональный** — решение принимает LLM, не пользователь и не код.
- **Decision-промпт делает дополнительный LLM-вызов** — каждый запрос пользователя порождает 2 вызова Ollama (decision + final).
- **Артефакты сборки в `src/`** (рядом с кодом). Удобно для одной машины, неудобно для контейнеризации.
- **Нет потоковой генерации** — пользователь ждёт полный ответ.

## 4. Что точно не сломано

Чтобы не паниковать без причины:

- FAISS-индекс корректно сериализуется и читается.
- `pypdf` и `python-docx` стабильны на типичных файлах (но могут давать пустой текст на сложно отформатированных PDF — это норма).
- Decision-промпт устойчив к мусору в JSON (есть fallback в `except`).
- `MCPClient` корректно завершает subprocess в `close()`.
- Корневой и `src/`-уровневые `.gitignore` покрывают типичные артефакты.
