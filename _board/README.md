# Доска задач (board)

Артефакты процесса для разработчика и/или LLM-агента: план задач, алгоритм выполнения, ad-hoc заметки. Документы здесь — **рабочее состояние**, в отличие от `../_docs/`, где лежит **спецификация и состояние проекта**.

## Состав

- [`plan.md`](./plan.md) — **индекс спринтов**: легенда статусов, правила, шаблон задачи, активный и закрытые спринты. Сами задачи спринта в этом файле не лежат.
- [`process.md`](./process.md) — пошаговый алгоритм выполнения **одной** задачи (универсален, не зависит от конкретной задачи).
- [`progress.txt`](./progress.txt) — ad-hoc заметки о текущем прогрессе и фиксация чек-листов приёмки.
- [`sprints/`](./sprints/) — каталог спринтов, **по одному файлу на спринт**: `sprints/<id>-<short-name>.md`. Внутри файла — ТЗ спринта, этапы и задачи (в формате шаблона из `plan.md`). История спринтов сохраняется: закрытые файлы остаются в каталоге без правок.

## Связи с другими артефактами

- Спецификация проекта (требования, архитектура, стек) — в `../_docs/`. Точка входа: `../_docs/README.md`.
- Поведенческие гайдлайны LLM-агента — `../CLAUDE.md` (общие, не проектные).
- Известные проблемы и легаси, которые могут влиять на задачу — `../_docs/current-state.md`.
- Кандидаты в следующие спринты — `../_docs/roadmap.md`.

## Точка входа для нового агента

```
../CLAUDE.md
   ↓
../_docs/README.md            (что это за проект)
   ↓
../_docs/current-state.md     (что не так уже сегодня)
   ↓
./README.md                   (этот файл)
   ↓
./plan.md                     (индекс спринтов)
   ↓
./sprints/<active>.md         (текущий спринт: ТЗ + задачи)
   ↓
./process.md                  (как выполнять одну задачу)
```

## Правила

1. **Одна задача в `Progress`**: на всей доске одновременно — **ровно одна** задача со статусом `Progress`.
2. **Один активный спринт**: в `plan.md` § «Индекс спринтов» — ровно один `Active`. Новый открывается только после закрытия предыдущего (или явно согласованного параллельного «фонового»).
3. **Берём сверху вниз** в файле активного спринта; нельзя брать задачу, если её зависимости (поле «Зависит от») не `Done`.
4. **Атомарные коммиты статуса**: каждая смена статуса фиксируется отдельным коммитом — `chore(plan): start task <sprint-id>.<task-id>` или `chore(plan): complete task <sprint-id>.<task-id>`.
5. **Definition of Done — обязательное**: задача переходит в `Done`, только если по каждому пункту DoD стоит «выполнено». Иначе — оставить `Progress` и доделать.
6. **Не выполнять смежные правки**: задача меняет только то, что в её описании. Всё, что заметили попутно — в `../_docs/current-state.md` или `../_docs/roadmap.md`.
7. **История спринтов неизменна**: закрытые файлы в `sprints/` не редактируются, кроме секции «История изменений» (даты, ссылки).

## Жизненный цикл задачи

```
ToDo  →  Progress  →  Done
    ↘             ↘
       Blocked    (или возврат в Progress, если DoD не закрыт)
```

- `ToDo` — готова к работе, Definition of Ready выполнен (известны все зависимости и связанные документы).
- `Progress` — взята в работу. **Только одна.**
- `Blocked` — заблокирована: нужны внешние действия (ТЗ, доступ, решение). В описании зафиксирована причина.
- `Done` — закрыта. Definition of Done выполнен полностью, изменения закоммичены и (если применимо) запушены.

## Источник новых спринтов и задач

1. **Из ТЗ пользователя** — после получения ТЗ создаётся новый файл `sprints/<id>-<name>.md` (спецификация + задачи) и обновляется индекс в `plan.md`.
2. **Из найденных багов** — запись попадает в `../_docs/current-state.md`. Если баг попадает в активный спринт — задача добавляется в его файл; иначе — в roadmap.
3. **Из roadmap** — `../_docs/roadmap.md` содержит кандидатов; конкретные задачи попадают в файл нового спринта при его планировании.

## Что НЕ делается на доске

- Не ведём «лог разработчика» (это место для `progress.txt`, и то ad-hoc).
- Не дублируем содержимое `../_docs/` — ссылаемся.
- Не оставляем «общие» задачи без чёткого DoD (не «исследовать что-то», а «исследовать и зафиксировать вывод в `current-state.md`»).
- Не редактируем закрытые файлы спринтов задним числом (кроме секции «История изменений»).
- Не складываем задачи активного спринта в `plan.md` — они живут только в `sprints/<id>-<name>.md`.
