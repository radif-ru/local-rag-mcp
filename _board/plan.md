# План — индекс спринтов

Документ — **оглавление** спринтов проекта. Сами задачи каждого спринта живут в отдельном файле `sprints/<id>-<short-name>.md`, чтобы история спринтов сохранялась.

## Состав файла

1. Легенда статусов (универсальная, для задач и спринтов).
2. Правила работы со спринтами и задачами.
3. Шаблон задачи (используется внутри файлов спринтов).
4. Индекс спринтов: активный + закрытые.
5. Фоновые / инфраструктурные задачи вне спринтов.

## Легенда статусов

### Задачи

- **ToDo** — задача готова к работе (Definition of Ready выполнен).
- **Progress** — задача в работе (взята исполнителем). Одновременно в `Progress` — **ровно одна** задача в рамках всей доски.
- **Done** — задача завершена, Definition of Done выполнен.
- **Blocked** — задача заблокирована (нужны внешние действия / разрешение зависимостей).

### Спринты

- **Active** — спринт идёт, по нему есть задачи в `Progress` и/или `ToDo`.
- **Closed** — все задачи спринта в `Done`, Acceptance Criteria закрыты.
- **Cancelled** — спринт остановлен до закрытия (фиксируется причина).

## Правила работы со спринтами

1. **Один активный спринт за раз.** Новый спринт открывается только после закрытия предыдущего. Исключение — заранее согласованный параллельный «фоновый» спринт (помечается явно в индексе).
2. **Каждому спринту — отдельный файл.** При старте создаётся `sprints/<id>-<short-name>.md` со структурой: цель, скоуп, acceptance criteria, этапы, задачи (по шаблону ниже), сводная таблица, риски, история изменений.
3. **История спринтов сохраняется.** Закрытые спринты остаются в `sprints/` без правок, только секция «История изменений» дополняется датой закрытия.
4. **Индекс обновляется синхронно.** При старте/закрытии спринта актуализируется секция «Индекс спринтов» в этом файле.
5. **Источник новых спринтов** — ТЗ от пользователя + кандидаты из `_docs/roadmap.md` и `_docs/current-state.md`.

## Правила работы с задачами

1. Задачи берутся **сверху вниз** в файле текущего спринта; нельзя брать новую, пока её зависимости (поле «Зависит от») не `Done`.
2. В `Progress` одновременно — только одна задача на всей доске.
3. При смене статуса — отдельный коммит:
   - `chore(plan): start task <sprint-id>.<task-id>` (например, `chore(plan): start task 01.1.2`),
   - `chore(plan): complete task <sprint-id>.<task-id>`.
4. Подробный процесс — `_board/process.md`.
5. Все коммиты спринта идут в его рабочую ветку (`feature/<short-name>` или `task-<id>-<name>`), указанную в шапке файла спринта.

## Шаблон новой задачи

Использовать как образец внутри файлов `sprints/<id>-<name>.md`:

```markdown
### Задача <sprint-id>.<task-id>. <Краткое название>

- **Статус:** ToDo
- **Приоритет:** high | medium | low
- **Объём:** XS | S | M | L | XL  (см. _docs/roadmap.md § «Размеры и риски»)
- **Зависит от:** Задача X.Y, Задача X.Z или —
- **Связанные документы:** `_docs/<file>.md` § <раздел>; ...
- **Затрагиваемые файлы:** `src/<path>.py`, ... (предполагаемые).

#### Описание

Что и зачем нужно сделать. Контекст, ссылки на источник (`_docs/current-state.md` § X.Y, либо запрос пользователя).

Конкретные шаги (если применимо):

1. Шаг 1.
2. Шаг 2.
3. ...

#### Definition of Done

- [ ] Условие 1 (проверяемо: smoke / тестом / инспекцией).
- [ ] Условие 2.
- [ ] Документация обновлена: `_docs/<file>.md` § <раздел>.
- [ ] (Если применимо) тесты добавлены / обновлены, `pytest -q` зелёный.
- [ ] `git status` чист, артефакты не закоммичены.
- [ ] Smoke-test пройден: `python main.py build-index` + 1–2 интерактивных вопроса.

---
```

## Фоновые / инфраструктурные задачи (вне спринтов)

Здесь фиксируются разовые chores, не относящиеся к фичам и не подпадающие под спринт. Их нумерация — сквозная, формат `Задача 0.<id>`.

### Задача 0.1. Документация и доска подготовлены

- **Статус:** Done
- **Приоритет:** high
- **Объём:** M
- **Зависит от:** —
- **Связанные документы:** `_docs/README.md`, `_board/README.md`.

#### Описание

Создать структуры `_docs/` и `_board/` с базовой документацией о существующем проекте: обзор, архитектура, стек, структура репо, RAG-конвейер, MCP, конфигурация, команды, правила разработки, текущее состояние и roadmap. Подготовить каркас доски (`_board/plan.md`, `_board/process.md`, `_board/progress.txt`).

#### Definition of Done

- [x] `_docs/README.md` со списком всех документов и порядком чтения.
- [x] `_docs/project.md` (обзор + философия).
- [x] `_docs/architecture.md` (компоненты + потоки данных).
- [x] `_docs/stack.md` (стек + версии + локальные требования).
- [x] `_docs/project-structure.md` (структура репо + правила размещения файлов).
- [x] `_docs/rag-pipeline.md` (детали ingest → chunk → embed → index → query).
- [x] `_docs/mcp-tools.md` (сервер, клиент, инструменты, безопасность, decision-протокол).
- [x] `_docs/configuration.md` (все параметры + пресеты).
- [x] `_docs/commands.md` (CLI-команды + типичные ошибки).
- [x] `_docs/instructions.md` (правила разработки).
- [x] `_docs/current-state.md` (известные баги и нюансы — не менее 10 записей).
- [x] `_docs/roadmap.md` (фазы развития).
- [x] `_board/README.md` (назначение доски и связь с `_docs/`).
- [x] `_board/process.md` (универсальный алгоритм работы над одной задачей).
- [x] `_board/plan.md` (этот файл) — индекс спринтов, легенда, шаблон задачи.
- [x] `_board/progress.txt` создан с заголовком и легендой.
