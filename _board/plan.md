# План — индекс спринтов

Документ — **оглавление** спринтов проекта. Сами задачи каждого спринта живут в отдельном файле `sprints/<id>-<short-name>.md`, чтобы история спринтов сохранялась.

## Состав файла

1. Легенда статусов (универсальная, для задач и спринтов).
2. Правила работы со спринтами и задачами.
3. Шаблон файла спринта (обязательные секции + декомпозиция «этапы → задачи»).
4. Шаблон задачи (используется внутри файлов спринтов).
5. Индекс спринтов: активный + закрытые.

## Легенда статусов

### Задачи

- **ToDo** — задача готова к работе (Definition of Ready выполнен).
- **Progress** — задача в работе (взята исполнителем). Одновременно в `Progress` — **ровно одна** задача в рамках всей доски.
- **Done** — задача завершена, Definition of Done выполнен.
- **Blocked** — задача заблокирована (нужны внешние действия / разрешение зависимостей).

### Спринты

- **Active** — спринт идёт, по нему есть задачи в `Progress` и/или `ToDo`.
- **Closed** — все задачи спринта в `Done`, Acceptance Criteria закрыты.
- **Cancelled** — спринт остановлен до закрытия (фиксируется причина).

## Правила работы со спринтами

1. **Один активный спринт за раз.** Новый спринт открывается только после закрытия предыдущего. Исключение — заранее согласованный параллельный «фоновый» спринт (помечается явно в индексе).
2. **Новый спринт = новая ветка.** Каждый новый спринт открывается в отдельной ветке `feature/<short-name>` от актуальной `main`. Имя ветки фиксируется в шапке файла спринта и в индексе спринтов. Коммиты спринта идут только в эту ветку; merge в `main` — после закрытия спринта. Исключение — спринт 00 (`main`, оформлен задним числом до введения этого правила).
3. **Каждому спринту — отдельный файл с фиксированной структурой.** При старте создаётся `sprints/<NN>-<short-name>.md` (с leading zero) — структура зафиксирована § «Шаблон файла спринта» ниже. **Декомпозиция:** спринт → этапы → задачи. Этапы нумеруются с 1, идут по порядку выполнения. Внутри этапа задачи нумеруются с 1 и имеют локальный ID `<stage>.<task>`. Зависимости задачи («Зависит от») могут указывать на задачи внутри этапа, на задачи других этапов того же спринта или на задачи закрытых ранее спринтов. Нельзя зависеть от задачи другого активного спринта (п.1).
4. **История спринтов сохраняется.** Закрытые спринты остаются в `sprints/` без правок, только секция «История изменений» дополняется датой закрытия.
5. **Индекс обновляется синхронно.** При старте/закрытии спринта актуализируется секция «Индекс спринтов» в этом файле.
6. **Источник новых спринтов** — ТЗ от пользователя + кандидаты из `_docs/roadmap.md` и `_docs/current-state.md`.

## Правила работы с задачами

1. Задачи берутся **сверху вниз** в файле текущего спринта; нельзя брать новую, пока её зависимости (поле «Зависит от») не `Done`.
2. В `Progress` одновременно — только одна задача на всей доске.
3. **Нумерация задач и формат коммитов:**
   - **Локальный ID** в файле спринта — двухчастный `<stage>.<task>`. `<stage>` — номер этапа в спринте (1, 2, 3, ...); `<task>` — номер задачи в этапе (1, 2, 3, ...). **Sprint-id в локальном ID НЕ дублируется.** Примеры: `1.1`, `2.3`, `3.6`.
   - **Коммит смены статуса** — формат `chore(plan): start task <NN>.<stage>.<task>` или `chore(plan): complete task <NN>.<stage>.<task>`, где `<NN>` — двузначный sprint-id с leading zero (совпадает с префиксом имени файла спринта). Примеры: `chore(plan): start task 02.1.4`, `chore(plan): complete task 03.2.1`.
4. Подробный процесс — `_board/process.md`.
5. Все коммиты спринта идут в его рабочую ветку (`feature/<short-name>` или `task-<id>-<name>`), указанную в шапке файла спринта.

## Шаблон файла спринта

Структура файла `sprints/<NN>-<short-name>.md` фиксирована. Обязательные секции, в указанном порядке:

```markdown
# Спринт <NN>. <Название>

- **Источник:** ТЗ пользователя / `_docs/roadmap.md` § ... / `_docs/current-state.md` § ...
- **Ветка:** `feature/<short-name>` (от `main`, см. § «Правила работы со спринтами», п.2).
- **Открыт:** YYYY-MM-DD
- **Закрыт:** — (или YYYY-MM-DD при закрытии)

## 1. Цель спринта

1–2 абзаца: чего достигаем и почему это важно сейчас.

## 2. Скоуп и non-goals

- **В скоупе:** явный список областей, которые трогаем.
- **Вне скоупа (non-goals):** явный список областей, которых НЕ касаемся, чтобы ограничить scope creep.

## 3. Acceptance Criteria спринта

Крупные проверяемые результаты, которые должны быть выполнены при закрытии спринта. AC спринта отличается от DoD отдельных задач: AC перекрывает эффект всего спринта целиком.

- [ ] Результат 1.
- [ ] Результат 2.
- [ ] Все задачи спринта — `Done`, сводная таблица актуальна.

## 4. Этап 1. <Название этапа>

1–3 строки о цели этапа.

### Задача 1.1. <Название задачи>

(См. § «Шаблон новой задачи» ниже.)

### Задача 1.2. <Название задачи>

...

## 5. Этап 2. <Название этапа>

### Задача 2.1. ...

## N. Этап K. <Название>

### Задача K.1. ...

## (N+1). Риски и смягчение

| # | Риск | Смягчение |
|---|------|------------|
| 1 | ...  | ...        |

## (N+2). Сводная таблица задач спринта

| #     | Задача | Приоритет | Объём | Статус | Зависит от |
|-------|---------|:---------:|:-----:|:------:|:----------:|
| 1.1   | ...     | ...       | ...   | ToDo   | —          |

> Обновляется при каждом переходе статуса и при добавлении/удалении задач.

## (N+3). История изменений спринта

- **YYYY-MM-DD** — спринт открыт, ветка `feature/<short-name>` создана от `main`.
- **YYYY-MM-DD** — закрыта задача X.Y: <кратко>.
- **YYYY-MM-DD** — спринт закрыт (или Cancelled с причиной).

---
```

**Правила файла спринта:**

- Секции 1–3 (Цель, Скоуп, AC) — пишутся при открытии спринта, в идеале не меняются.
- Секции 4..N (Этапы) — обязательно минимум один этап, у каждого этапа — отдельная `##`-секция, минимум одна задача внутри.
- Секции (N+1)..(N+3) — обязательные вне зависимости от размера спринта.
- **Задачи внутри этапа** — оформляются по § «Шаблон новой задачи» (ниже).
- **Порядок этапов** отражает порядок выполнения. Нельзя начинать задачи этапа K, пока не закрыты все блокирующие зависимости из этапов < K.
- **Новые задачи можно добавлять** в активный спринт по ходу работы — это не нарушает «история неизменна» (запрет касается **закрытых** спринтов).
- **Новые задачи получают следующий свободный ID** в своём этапе; номера ранее закрытых задач не переиспользуются.
- **Закрытие спринта:** все задачи — `Done`, все пункты AC проставлены, запись в § «История изменений» + индекс спринтов в этом файле обновлены.

## Шаблон новой задачи

Использовать как образец внутри файлов `sprints/<id>-<name>.md`:

```markdown
### Задача <stage>.<task>. <Краткое название>

- **Статус:** ToDo
- **Приоритет:** high | medium | low
- **Объём:** XS | S | M | L | XL  (см. _docs/roadmap.md § «Размеры и риски»)
- **Зависит от:** Задача X.Y, Задача X.Z или —
- **Связанные документы:** `_docs/<file>.md` § <раздел>; ...
- **Затрагиваемые файлы:** `src/<path>.py`, ... (предполагаемые).

#### Описание

Что и зачем нужно сделать. Контекст, ссылки на источник (`_docs/current-state.md` § X.Y, либо запрос пользователя).

Конкретные шаги (если применимо):

1. Шаг 1.
2. Шаг 2.
3. ...

#### Definition of Done

- [ ] Условие 1 (проверяемо: smoke / тестом / инспекцией).
- [ ] Условие 2.
- [ ] Документация обновлена: `_docs/<file>.md` § <раздел>.
- [ ] (Если применимо) тесты добавлены / обновлены, `pytest -q` зелёный.
- [ ] `git status` чист, артефакты не закоммичены.
- [ ] Smoke-test пройден: `python main.py build-index` + 1–2 интерактивных вопроса.

---
```
