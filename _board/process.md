# Процесс выполнения одной задачи

Документ описывает пошаговый алгоритм для закрытия **одной** задачи из `_board/plan.md`. Алгоритм универсален — не зависит от конкретной задачи. Содержание задачи, её Definition of Done и связанные документы берутся из `_board/plan.md` и `_docs/`.

## 0. Перед началом сессии

Один раз на сессию работы:

1. `git status` пуст, ветка актуальна (`git pull`).
2. Активировано виртуальное окружение: `source .venv/bin/activate`.
3. Зависимости актуальны: `pip install -r src/requirements.txt` (если был добавлен/обновлён модуль).
4. Ollama запущена: `ollama list` отвечает.
5. (При необходимости) индекс собран: `cd src && python main.py build-index`.

## 1. Прочитать документацию

**Обязательный шаг перед началом любой задачи.**

1. `CLAUDE.md` — поведенческие гайдлайны LLM-агента (think before coding, simplicity, surgical changes, goal-driven). Применяются к любой задаче.
2. `_docs/README.md` — навигация.
3. `_docs/current-state.md` — есть ли уже запись про затрагиваемую область (потенциальные подводные камни).
4. `_docs/instructions.md` — правила разработки. Перечитывать до автоматизма.
5. `_board/plan.md` — выбираемая задача и её соседи (зависимости).
6. Документы из поля «Связанные документы» выбранной задачи (например, `_docs/rag-pipeline.md` для задач по RAG).

При расхождении документа и кода — приоритет у кода. Документ обновится после задачи отдельным коммитом.

## 2. Выбрать задачу

1. Открыть `_board/plan.md`.
2. Найти **первую сверху** задачу со статусом `ToDo`.
3. Проверить поле «Зависит от»: все указанные задачи должны быть `Done`. Если нет — взять предыдущую (по зависимостям, не по порядку).
4. Если требуются внешние действия (ТЗ, доступы, решения от пользователя), которых нет — пометить задачу `Blocked` с причиной, остановиться.

## 3. Перевести задачу в статус `Progress`

1. В `_board/plan.md` у выбранной задачи изменить `**Статус:** ToDo` → `**Статус:** Progress`.
2. Обновить сводную таблицу в конце `_board/plan.md` (если есть).
3. Закоммитить:
   ```bash
   git add _board/plan.md
   git commit -m "chore(plan): start task <N> — <краткое название>"
   ```
4. (Опционально) создать ветку:
   ```bash
   git checkout -b task-<N>-<kebab-case>
   ```

> Правило: в статусе `Progress` одновременно — **ровно одна** задача.

## 4. Подготовить контекст для реализации

1. Прочитать «Описание» и «Definition of Done» задачи целиком.
2. Открыть в редакторе файлы из секции «Затрагиваемые файлы» (если указана) или предполагаемые по описанию.
3. Если задача связана с конкретной подсистемой — открыть параллельно соответствующий документ из `_docs/` (`rag-pipeline.md` / `mcp-tools.md` / `commands.md` / `configuration.md`).
4. Сформулировать (для себя или для LLM-помощника) краткий план шагов.

## 5. Реализовать задачу

Следуем `_docs/instructions.md` § 2 (минимализм, хирургические правки, цель → проверка):

1. **Самый минимальный диф**, решающий проблему.
2. **Не править соседний код** без явной задачи на это.
3. **Не добавлять «гибкости на будущее»** — реализуй ровно то, что нужно.
4. **Соблюдай существующий стиль** (импорты, нейминг, структура файлов).
5. Если встретился несвязанный баг или мёртвый код — записать в `_docs/current-state.md`, **не править** в этой задаче.

Цикл «правка → проверка»:

1. Внести правку.
2. Прогнать smoke-проверки (см. § 6).
3. Если что-то поломалось — откатить и переделать.
4. Если зафиксированы новые тесты — добавлять/обновлять до зелёного состояния.

## 6. Проверка изменений

### 6.1 Минимальный smoke-test (всегда)

```bash
# 1. Импорт пакетов не сломан
cd src
python -c "from assistant import CompanyKBAssistant; print('OK')"

# 2. Сборка индекса (если затронут rag/* или config)
python main.py build-index

# 3. Один интерактивный вопрос
python main.py
# ❓ Question: <тестовый вопрос по документам>
# ...проверяем ответ + источники + флаг MCP...
# exit
```

### 6.2 Целевые проверки (когда применимо)

- **MCP-инструмент** изменён — ручной вызов через `MCPClient` (`_docs/mcp-tools.md` § «Тестирование MCP вручную»).
- **Чанкование/эмбеддинги** изменены — пересобрать индекс, сравнить число чанков и качество ответов на 3 типичных вопроса.
- **Конфигурация** — `python -c "from config import *; print(...)"` для проверки значений.
- **MCP-сервер** — запустить вручную (`python src/mcp/server.py`), отправить руками `initialize` + `tools/call`.

### 6.3 Когда появятся pytest

```bash
pytest -q                                      # все тесты
pytest tests/<затронутый-модуль> -q -v         # детально по подсистеме
pytest --cov=src --cov-report=term-missing     # покрытие
```

## 7. Проверить Definition of Done

1. Открыть в `_board/plan.md` раздел Definition of Done выбранной задачи.
2. По каждому пункту — отметить «выполнено / не выполнено» (можно прямо в `progress.txt`, если задача крупная).
3. Если хотя бы один пункт не закрыт — вернуться к §5.
4. Если в DoD есть «обновить документацию X» — обновить и проверить, что `_docs/README.md` (раздел «Навигация») и связанные ссылки всё ещё корректны.
5. Прогнать `git status` — никаких лишних файлов (особенно `*.faiss`, `*.pkl`, `.env`, документов из `src/docs/`).

## 8. Зафиксировать результат в git

Коммиты — атомарные, conventional commits, в императиве:

```bash
git add <изменённые файлы>
git commit -m "feat(<scope>): <что сделано>"
# или: fix(...) / refactor(...) / docs(...) / test(...) / chore(...)
```

Примеры (типичные scope этого проекта):

- `feat(rag): add chunk overlap configuration validation`
- `fix(ingest): remove dead else-branch in document loader`
- `refactor(assistant): extract decision-prompt builder`
- `docs(architecture): clarify mcp tool decision flow`
- `test(rag): cover chunk_text deterministic output`
- `chore(plan): complete task 3`

## 9. Перевести задачу в статус `Done`

1. В `_board/plan.md` изменить `**Статус:** Progress` → `**Статус:** Done`.
2. Обновить сводную таблицу.
3. Коммит:
   ```bash
   git add _board/plan.md
   git commit -m "chore(plan): complete task <N>"
   ```
4. Пуш (или открыть PR, если работали в ветке):
   ```bash
   git push
   ```

## 10. Обновить документацию (если нужно)

Если в ходе задачи изменилось поведение, отражённое в `_docs/`:

1. Найти соответствующий документ (`commands.md`, `architecture.md`, `configuration.md`, `rag-pipeline.md`, `mcp-tools.md`).
2. Внести правку.
3. Если изменилась структура каталога или появился новый документ — обновить `_docs/README.md` § «Навигация».
4. Коммит `docs(<scope>): ...`.

## 11. Перейти к следующей задаче

1. Вернуться к §2 — выбрать следующую задачу `ToDo`.
2. Если все задачи `Done`, а `_board/plan.md` пуст — спринт закрыт. Прогнать чек-лист приёмки (если есть в `progress.txt` или в задаче §10), отчитаться пользователю.

---

## Короткая шпаргалка (TL;DR)

1. Прочитать `CLAUDE.md` → `_docs/README.md` → `_docs/current-state.md` → `_board/plan.md` → документы задачи.
2. Взять верхнюю `ToDo` задачу — проверить, что зависимости `Done`.
3. Сменить статус на `Progress`, коммит `chore(plan): start task N`.
4. Минимальный диф → smoke-test → зелёный.
5. По каждому пункту DoD — отметка «выполнено».
6. Коммит правок (`feat/fix/refactor/docs/test/chore`).
7. Сменить статус на `Done`, коммит `chore(plan): complete task N`, пуш.
8. К следующей задаче.

## Чек-лист перед переходом к следующей задаче

- [ ] Definition of Done закрыт по всем пунктам.
- [ ] Smoke-test пройден (build-index + интерактивный вопрос).
- [ ] Если есть pytest — `pytest -q` зелёный.
- [ ] `git status` чист (нет лишних артефактов / документов / `.env`).
- [ ] Документация обновлена там, где меняется поведение.
- [ ] Статус задачи в `_board/plan.md` = `Done`, сводная таблица актуальна.
- [ ] Коммиты запушены.
